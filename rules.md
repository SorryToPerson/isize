# iSize 项目开发规范

本文档用于约束 `isize` 项目的多端研发方式。当前项目目标是构建一个本地优先的图标处理产品，支持用户上传图片、裁切为正方形、按平台规范批量生成图标，并导出压缩包或单个文件。

---

## 1. 项目定位与不可违背的约束

### 1.1 产品定位

- `isize` 是一个通用图标处理工具，优先覆盖 `WEB / APP / 小程序 / 桌面应用图标` 等输出场景。
- 产品核心价值不是“在线存图”，而是“快速、准确、低门槛地生成多平台图标规格”。

### 1.2 强约束

- **本地优先**: 所有项目记录、历史记录、最近导出配置默认只保存在客户端本地。
- **服务端无持久化**: 后端只做兜底处理与临时计算，不保存用户原图、处理结果和历史记录。
- **客户端优先处理**: 能在 `WEB / APP` 客户端完成的裁切、缩放、压缩和打包，不交给后端。
- **平台规范优先**: 图标尺寸、命名和导出结构必须以平台官方规范为准，UI 体验不能覆盖规范正确性。
- **无账号依赖**: MVP 阶段不引入登录、云端项目同步和在线素材库。

---

## 2. 仓库结构规范

### 2.1 工作区结构

- 根目录使用 **npm workspaces** 管理。
- 标准目录：
  - `apps/web`: Web 端，基于 React + Vite。
  - `apps/app`: App 端，基于 React Native；MVP 阶段采用 Expo 管理工作流以降低启动成本。
  - `apps/server`: 后端服务，基于 NestJS。
  - `packages/contracts`: 跨端共享的规格、类型、常量与产品约束。
  - `docs`: PRD、架构说明、阶段规划等文档。

### 2.2 模块职责

- `web`:
  - 负责图片上传、裁切、平台选择、生成预览、导出历史。
  - 优先实现浏览器内图像处理。
- `app`:
  - 负责移动端拍照/相册导入、裁切、移动端导出与分享。
  - 保持和 Web 端一致的业务流程与术语。
- `server`:
  - 只承担规则查询、任务规划、重型图像处理兜底与压缩输出。
  - 默认设计为无状态服务。
- `contracts`:
  - 存放平台规格、导出命名、通用类型、处理策略枚举。
  - 新增平台能力时必须先更新这里。

## 3. 通用工程规范

### 3.1 语言与类型

- 全仓默认使用 **TypeScript**。
- 严禁无边界地使用 `any`；必须优先定义接口、联合类型、枚举或字面量约束。
- 对外 API、导出任务、平台规格、历史记录结构都必须显式建模。
- 涉及平台枚举时，优先使用字符串字面量联合，不要在多个应用中散落硬编码。

### 3.2 命名规范

- 文件与目录：使用 `kebab-case`。
- 变量、函数：使用 `camelCase`。
- React 组件、NestJS 类、DTO、Service：使用 `PascalCase`。
- 常量：使用 `UPPER_SNAKE_CASE`。
- 文档文件：采用明确语义命名，如 `PRD.md`、`architecture.md`、`milestones.md`。

### 3.3 代码组织

- 页面只负责组合模块，不承载大段业务逻辑。
- 图像处理、导出编排、平台规格匹配必须提取到独立服务层或 hooks。
- 单文件过长时优先拆成：
  - `components`
  - `hooks`
  - `services`
  - `types`
  - `constants`

### 3.4 文档同步

- 新增平台规格时，必须同步更新：
  - `packages/contracts`
  - `docs/PRD.md`
  - 相关端的交互文案或导出说明
- 产品流程发生变化时，先改 PRD，再改实现。

## 4. WEB 端规范

### 4.1 技术基线

- 使用 **React + Vite + TypeScript**。
- UI 优先保持轻量，不预设大型 UI 框架；如引入组件库，必须说明理由。

### 4.2 图像处理规范

- 优先使用浏览器能力完成：
  - `Canvas / OffscreenCanvas`
  - `ImageBitmap`
  - `Web Worker`
  - 前端压缩与 `zip` 打包
- 重型任务才允许回退后端：
  - 超大尺寸原图
  - 浏览器内存不足
  - 某些格式转换受限
  - 多平台复杂打包策略

### 4.3 状态管理

- 单页范围状态优先使用 React 内建状态。
- 只有在跨页面共享处理任务、历史记录、平台配置时，才引入集中式状态方案。
- 本地历史记录统一封装在持久化层，禁止页面中直接散落 `localStorage` 调用。

---

## 5. APP 端规范

### 5.1 技术基线

- 使用 **React Native + TypeScript**。
- 当前仓库骨架采用 **Expo 管理工作流**，用于加快 MVP 联调和视觉验证。
- 若后续接入更深的华为设备能力、原生分享能力或高性能图像处理库，可切换到 Bare Workflow，但业务模块目录结构保持不变。

### 5.2 平台兼容规范

- 不直接依赖浏览器专有 API。
- 文件访问、分享、图片选择、缓存路径必须通过统一的适配层封装。
- 和华为相关的差异逻辑单独收敛在平台服务中，不允许散落在页面组件内。

### 5.3 交互原则

- App 端流程要比 Web 更短，优先突出：
  - 拍照/相册导入
  - 一步裁切
  - 快速选择预设
  - 直接保存/分享
- 不在首屏堆叠复杂配置，复杂平台参数放到“高级选项”。

---

## 6. 服务端规范（NestJS）

### 6.1 职责边界

- 服务端只负责：
  - 输出规格查询
  - 图像任务规划
  - 重型处理兜底
  - 压缩包构建
- 服务端不负责：
  - 用户账号
  - 永久存储
  - 项目历史
  - 图片素材管理

### 6.2 API 设计

- 所有接口默认走 `/api` 前缀。
- 输入参数必须通过 `DTO + class-validator` 校验。
- 响应结构保持稳定，避免直接透出底层处理库的原始结构。
- 导出接口需明确同步/异步策略，避免前端轮询模型后续难以演进。

### 6.3 临时文件与性能

- 若后端生成临时文件，必须有明确清理机制。
- 压缩、缩放、格式转换等 CPU 密集任务必须可单独迁移，不与普通配置接口混在同一模块。

---

## 7. 图标规格与导出规范

### 7.1 输入图规范

- 默认推荐上传 `1024 x 1024` 或更高分辨率原图。
- 裁切流程必须以正方形为核心，不支持自由比例作为默认主流程。
- 对非透明背景、圆角预处理等风险要提前提示用户。

### 7.2 输出规格规范

- 每个平台的尺寸、命名、目录结构单独建模，不允许只维护一个“尺寸数组”。
- 需要区分：
  - 显示尺寸
  - 导出尺寸
  - 文件名
  - 使用场景
  - 是否建议客户端处理
- iOS / macOS / Android / Windows 的图标圆角、透明区、安全区提示必须由规则层提供，而不是只写在文案里。

### 7.3 导出规范

- 支持单个导出与批量导出。
- 批量导出默认输出 zip。
- 导出文件命名遵循：
  - `platform/size/fileName`
  - 或面向平台的约定目录结构
- 同一批次导出要保留任务快照，便于本地历史回放。

---

## 8. AI 协作规范

当使用 AI 生成代码、文档或方案时，必须显式带上以下约束：

- “这是一个 React + React Native + NestJS 的多端图标处理项目。”
- “项目是本地优先架构，服务端不做持久化存储。”
- “图像处理默认客户端优先，后端只做兜底。”
- “新增平台规格时要同步更新 contracts、PRD 和导出逻辑。”
- “输出必须优先考虑平台规范正确性，而不是只追求页面展示效果。”

---

## 9. Git 与提交规范

- 提交信息遵循 **Conventional Commits**。
- 推荐使用：
  - `feat`: 新功能
  - `fix`: 缺陷修复
  - `docs`: 文档更新
  - `refactor`: 重构
  - `style`: 纯样式或格式调整
  - `chore`: 脚手架、依赖、构建配置
- 大功能分多次提交时，先提交结构与文档，再提交具体业务实现。
